# Pear — AI Assistant Guide

## What This App Is

Pear (package `hane`, slug `wingmate`) is an iOS dating app. Two roles:

- **Dater** — browses profiles, swipes, chats with matches.
- **Winger** — a trusted friend who swipes on behalf of a dater.

Core flows: Auth → Onboarding → Discover → Matches → Messaging → Wingpeople.

## How to Add a New Feature

1. **SQL migration** in `supabase/migrations/<timestamp>_<feature>.sql` — always enable RLS.
2. **`npm run codegen`** — regenerates `types/database.ts`, `db/schema.ts`, `openapi.json`, `lib/api/generated/`.
3. **Domain** under `supabase/functions/api/domains/<feature>/`:
   - `schemas.ts` — Zod shapes tagged `.openapi(name)`. Derive enum literals from `pgEnum.enumValues`.
   - `queries.ts` — Drizzle fetches; first arg always `db: DBOrTx`. No Hono/Zod imports.
   - `transformers.ts` — snake_case → camelCase row mappers.
   - `route.ts` — `createRoute` + `mount<Feature>(app)`. Use `getDeps(c)` for `{ db, userId, push }`.
   - Register `mount<Feature>(app)` in `app.ts`.
4. **`npm run codegen`** again to pick up the new route.
5. **Screen** — import generated hook: `useGetApi<Feature>Suspense()` from `@/lib/api/generated/<tag>/<tag>.ts`. Destructure `data` directly — no wrapper to peel.
6. **Mutations** — call the generated function directly: `await postApi<Feature>(body)` returns the success body or throws. Wire with `useMutation` + `invalidateQueries`.
7. **Commit** generated artifacts in the same commit as the source that produced them.

## Routing & Auth Gate

`RootNavigator` in `app/_layout.tsx` reads `session` from `useSession()`:

- No session → `/(auth)/login`
- No `chosen_name` → `/(onboarding)`
- Role = `winger` → `/(winger-tabs)`
- No `datingProfile` → `/(onboarding)`
- Otherwise → `/(tabs)/discover`

Two `useEffect`s allowed in `AuthenticatedNavigator`: deep-link invite check from AsyncStorage, and push token registration — both genuine external events.

## Query Patterns

All data fetching goes through Orval-generated hooks. `lib/api/http.ts` returns the parsed body directly — no `{ data, status }` wrapper, no status checks at callsites.

For Supabase realtime/auth/storage, use the supabase-js client directly.

Don't add new RPCs or PostgREST selects from the client — add a Hono route instead.

## Codegen

```bash
npm run codegen
# supabase gen types typescript   → types/database.ts
# drizzle-kit introspect          → supabase/functions/api/db/schema.ts
# deno run emit-spec.ts           → openapi.json
# orval                           → lib/api/generated/
```

Run after: SQL migrations, new/changed Hono routes, Zod schema changes. Idempotent — when in doubt, run it.

**Hard constraints:**

- Never hand-edit `types/database.ts` or `db/schema.ts`. Regenerate.
- Never run `drizzle-kit generate` or `drizzle-kit push`. Schema writes go through SQL migrations only.
- After `db:drizzle` introspects, strip the `profiles.id → auth.users` FK (not in `public` schema).
- Generated artifacts belong in the same commit as the source change.

## Edge Functions Architecture

`supabase/functions/api/` is a single Hono app. Key points:

- **Auth middleware** sets `c.var.{userId, token, claims}` from JWT. Kong verifies in prod; skipped locally with `--no-verify-jwt`.
- **Transaction middleware** opens a Drizzle tx per request, sets `role = authenticated` + `request.jwt.claims` for RLS, exposes as `c.var.db`. Commits on `return`, rolls back on `throw`. A returned 4xx **commits**; a thrown `HTTPException` rolls back.
- **Never import the module-level `db`** from `db/client.ts` in handlers — always use `c.var.db` via `getDeps(c)`.
- **Push** via `c.var.push.send(token, title, body)`. Storage via `c.var.supabase`. No new standalone edge functions — side effects belong in Hono handlers.
- **RLS is the auth floor.** Handler-side `where(eq(...))` clauses are for correctness/perf, not security.
- **Secrets:** `DATABASE_URL` (authenticator role, Supavisor in prod, `host.docker.internal:54322` locally). `SUPABASE_URL/ANON_KEY/REGION` auto-injected. `SUPABASE_REGION` absence = local dev (push no-ops).

## Styling (NativeWind v5 + Tailwind v4)

- **Always** import styled primitives from `@/lib/tw`: `View`, `Text`, `Pressable`, `ScrollView`, `TextInput`, `SafeAreaView`, `AnimatedView`.
- Never use `StyleSheet.create` — use `className`.
- `style` prop only for: `borderCurve`, dynamic/animated values, SVG props.
- Never use function-style `style` on `Pressable` (NativeWind bug #1105 — children don't render). Use `active:` pseudo-class.
- `cn()` from `@/lib/cn` for conditional classes.
- Design tokens in `global.css` under `@theme` — never add raw hex to `constants/theme.ts`.

**Modal color caveat:** CSS variables aren't injected in Modal's native layer. Use `ModalView` from `@/lib/tw` with a `backgroundColor` style prop (not className) for colors inside Modals.

## Coding Preferences

- **Composition:** Orchestrators decide what to render; logic lives close to where it's used.
- **Control flow:** `switch` on discriminated unions. Explicit `return` on every branch.
- **No `useEffect`:** Move async work into transition handlers. Exceptions: Supabase auth subscription, push token registration, AsyncStorage deep-link check (mount-only, genuine external events).
- **Error propagation:** No try/catch across callback boundaries. Return errors as values. User-facing errors via `toast.error()`.
- **Forms:** react-hook-form everywhere — `Controller`, `handleSubmit`, `isSubmitting`, `isValid`, `mode: 'onChange'`.
- **Queries:** Transforms belong in the query function. Unwrapping boilerplate belongs in the query wrapper, not callsites. No magic string cache keys.
- `async/await` over `.then()`.
